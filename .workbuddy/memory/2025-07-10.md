# 2025-07-10 工作日志

## 代码审查：货运财务系统

对货运财务系统（Next.js 16 + Drizzle ORM + SQLite）进行了全面代码审查，通读了数据库 schema、全部 API 路由（33个）、核心 lib 文件、主要前端页面。

### 发现的主要问题（按严重程度）

**P0 - 金额单位全系统不统一**
- `_cents` 后缀字段实际存储单位混乱：LoadingExpenseManager 里除以100（分→元），但几乎所有页面直接 `.toFixed(6)` 显示分值
- 同一页面（如 loading-lists/[id]）单价列除以100，应收列不除，金额差100倍
- 仪表盘混用 formatCents（除100）和直接显示，数字不一致

**P0 - 仪表盘/月报营收与成本计算逻辑错误**
- 装柜「需支付总价」同时计入营收和成本
- 拼柜营收用 `客户应收_cents` 但过滤条件是 `cost_status !== '待支出'`
- 成本无状态过滤，所有当月创建的都算成本

**P0 - 账单金额两套计算逻辑不一致**
- bills POST 用 `需支付总价_cents`
- bills export 和 ai/classify 用价格矩阵×计费体积重新算

**P1 - 账单重新生成不重置付款状态**
**P1 - customers GET 无权限检查**
**P1 - upload 无文件类型/大小限制，上传到 public 目录**
**P1 - 金额字段混用 real 和 integer 类型**

**P2 - metrics.ts 月均票数没除以6，overdueCount 恒为0**
**P2 - 自动创建客户用唛头号当客户名**
**P2 - 批量付款按 markId 更新可能误标记其他批次**
**P2 - expenses PUT 全量覆盖字段**
**P2 - 数据库路径硬编码 data.db**
**P2 - DeepSeek 模型名可能错误**
**P2 - 账单模板合计行写死"泰铢"**

## 修复执行

确认数据库无正式数据（data.db 是旧版空库），修改零风险。确定 `_cents` 字段大部分代码按「元」处理，只有少数地方错误地多乘/多除了 100。

### 已完成修复

**安全漏洞：**
- customers GET 加了 session 验证
- bills PATCH 加了 viewer 角色拦截
- upload 加了文件类型限制（仅图片+PDF）和 10MB 大小限制，扩展名不再从用户输入取

**金额单位统一（统一为「存元直接显示」）：**
- format.ts: formatCents 去掉 /100，centsToYuan 去掉 /100，小数位 6→2
- LoadingExpenseManager.tsx: 去掉读取时 /100 和保存时 *100
- loading-lists/[id]/page.tsx: 单价列去掉 /100
- ScItemEditDialog.tsx: 读取时去掉 /100
- ai/verify-loading, loading-batches/ai-verify, shared-containers/ai-verify: 去掉 /100
- ai/extract-loading: receivable 计算去掉 /100

**仪表盘/月报计算逻辑：**
- 仪表盘: 营收去掉 cost_status/payment_status 条件；成本去掉装柜需支付总价（它是收入不是成本）；expenses 日期过滤从 paidDate 改成 createdAt
- 月报: 同样修复，成本去掉装柜需支付总价，expenses 统一用 createdAt

**账单逻辑：**
- bills POST: 拼柜金额从 `需支付总价_cents` 改成 `客户应收_cents`；重新生成时重置 paidAmount/remainingAmount/paymentStatus/paidAt
- bills export: receivable 优先用导入值（拼柜用客户应收，装柜用需支付总价），fallback 才用价格矩阵
- ai/classify: billNo 加 monthTag 避免冲突；customerId 为 0 时跳过；重新生成时重置付款状态
- bills pay: paymentStatus 为 undefined 时不写入；已付款时自动算 paidAmount

**其他修复：**
- metrics.ts: monthlyShipments 除以 6；overdueCount 实际计算（创建超60天且无回款）；评级逻辑考虑 overdueCount
- expenses PUT: 改成只更新传了的字段
- ai.ts: 模型名从 deepseek-v4-pro 改成 deepseek-chat
- generate-bill-xlsx.ts: 合计行币种不再写死"泰铢"

## 新增 AI 验价功能
- 新增 `/api/ai/verify-items` 接口：直接接收解析后的 items 数据，发给 DeepSeek 逐条检查，返回 verdict/reason
- 修改拼柜上传页面：解析完表格后自动调 verify-items，预览页面显示"AI 验价中..."提示，AI 结果出来后更新 verdict/reason
- 装柜目前只有手动录入没有上传解析，暂未加装柜 AI 验价
- 原有三个验价路由（ai/verify-loading、loading-batches/ai-verify、shared-containers/ai-verify）仍是死代码，ai/verify-loading 与 loading-batches/ai-verify 重复

## AI 验价改为按运单分组分批

用户反馈通常上千条数据，单次请求会超出 DeepSeek 64K 上下文窗口。改为：
- verify-items 接口按运单号分组，同一个运单的明细不拆开
- 每批最多 400 条或 50 个运单，确保不超窗口
- 提示词改成只返回异常项，通过的不返回（减少输出 token）
- 前端发送数据时带上运单号，结果处理改成：AI 没提到的默认"通过"，提到的标"异常"
- 2000 条数据约分 5 批，DeepSeek 不用换模型

## 导入时自动算客户应收

用户明确：Excel 里的价格是成本价（付给供应商的），向客户收的钱（收入）应该用价格矩阵重新算，不能用成本价当收入。

### 改动
- shared-container-items/route.ts：导入拼柜数据时，查客户价格矩阵，根据运输方式+货型确定单价，算计费体积（含最低消费），算出客户应收存到数据库
- ai/classify/route.ts：金额计算改成优先用数据库里的"客户应收"（拼柜）或"需支付总价"（装柜），没有才用价格矩阵算 fallback；mode 根据 type 判断写"拼柜"或"装柜"

### 测试结果
- 陶瓷杯：海运普货 300 × 体积 2.0 = 客户应收 600 ✓
- 塑料玩具：海运商检 450 × 体积 1.5 = 客户应收 675 ✓
- 导入时不传客户应收，系统自动算好存入数据库 ✓
